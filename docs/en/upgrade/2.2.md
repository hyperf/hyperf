# 2.2 Upgrade Guide

Version 2.2 mainly adds support for `PHP 8` and native Annotations.

## Modify Hyperf Component Versions

Directly change all `hyperf/*` in `composer.json` to `2.2.*`.

> `hyperf/engine` does not follow the framework version number, so it does not need to be modified.

In addition, we can execute `composer require "hyperf/ide-helper:2.2.*" --dev` to install `hyperf/ide-helper`. This component can help us prompt for parameters that can be set in Annotations when using native Annotations.

After that, just execute `composer update -o` to complete the upgrade normally.

## Modify Unit Test Script

Add the option `--prepend test/bootstrap.php`:

```json
{
    "scripts": {
        "test": "co-phpunit --prepend test/bootstrap.php -c phpunit.xml --colors=always"
    }
}
```

## Install pcntl Extension

The new version of Annotation scanning uses the `pcntl` extension, so please ensure that your `PHP` has this extension installed.

```shell
php --ri pcntl

pcntl
pcntl support => enabled
```

> When `grpc` is enabled, you need to add `grpc.enable_fork_support= 1;` to `php.ini` to support enabling child processes.

## AMQP

> Users using AMQP, please note that those who do not use it can ignore this section.

Because the `AMQP` component has been fully upgraded to support multiplexing, there are also some changes in configuration. Please modify as appropriate according to the latest configuration below.

```php
<?php

return [
    'default' => [
        'host' => env('AMQP_HOST', 'localhost'),
        'port' => (int) env('AMQP_PORT', 5672),
        'user' => env('AMQP_USER', 'guest'),
        'password' => env('AMQP_PASSWORD', 'guest'),
        'vhost' => env('AMQP_VHOST', '/'),
        'concurrent' => [
            'limit' => 1,
        ],
        'pool' => [
            // Number of connections opened at the same time
            // Because the new version of connection supports multiplexing, you can achieve high concurrency with very few connections
            'connections' => 2,
        ],
        'params' => [
            'insist' => false,
            'login_method' => 'AMQPLAIN',
            'login_response' => null,
            'locale' => 'en_US',
            'connection_timeout' => 3,
            'read_write_timeout' => 6,
            'context' => null,
            'keepalive' => true,
            'heartbeat' => 3,
            'channel_rpc_timeout' => 0.0,
            'close_on_destruct' => false,
            // Maximum number of idle Channels in multiplexing, exceeding this number will close excess limited Channels
            'max_idle_channels' => 10,
        ],
    ],
];
```

## Configuration Center

> Users using Configuration Center, please note that those who do not use it can ignore this section.

The Configuration Center has been completely refactored in this version. Please be sure to carefully re-read the corresponding documentation.

You need to introduce the `hyperf/config-center` component. The command is as follows:

```shell
composer require "hyperf/config-center:~2.2.0"
```

And introduce the corresponding driver dependency component according to the driver used. For example, if you use `Apollo`, you need to introduce the `hyperf/config-apollo` component. Other drivers are similar.

At the same time, all configuration information related to the configuration center has been collected in `config/autoload/config_center.php`. Please configure it according to the new configuration structure. If this file does not exist, you can create it by executing the `php bin/hyperf.php vendor:publish hyperf/config-center` command.

## Service Center

Users using the `hyperf/service-governance` component: because the `consul` adapter has been stripped from this component, you need to additionally introduce the `hyperf/service-governance-consul` component in the new version. The command is as follows:

```shell
composer require "hyperf/service-governance-consul:~2.2.0"
```

Users using `nacos` as the service center driver need to introduce the `hyperf/service-governance-nacos` component. The command is as follows:

```shell
composer require "hyperf/service-governance-nacos:~2.2.0"
```

## php-cs-fixer

If you do not need to upgrade `php-cs-fixer` to version `3.0`, you can ignore this section.

1. Modify version number

```
"friendsofphp/php-cs-fixer": "^3.0"
```

2. Rename .php_cs file

Rename to `.php-cs-fixer.php` and modify the corresponding code according to the following change record:

```diff
- return PhpCsFixer\Config::create()
+ return (new PhpCsFixer\Config())

- 'commentType' => 'PHPDoc',
+ 'comment_type' => 'PHPDoc',
```

## Compatibility of DI decreased in PHP 7.3

> Users using higher versions of PHP can ignore this section.

In versions `2.0` - `2.1`, to implement `AOP` on objects not managed by `DI` (such as objects instantiated by the `new` keyword), the underlying implementation used the `BetterReflection` component to implement related functions. While bringing a new programming experience, it also brought some problems that were difficult to overcome, such as:

- Project startup is very slow when there is no scan cache
- `Inject` and `Value` do not take effect in special scenarios
- `BetterReflection` does not yet support PHP 8 (as of v2.2 release)

In the new version, the application of `BetterReflection` is deprecated, and `sub-process scanning` is adopted to solve these pain points. However, there are some incompatibilities in lower versions of `PHP`:

After using `PHP 7.3` to start the application, you may encounter errors similar to the following:

```bash
PHP Fatal error:  Interface 'Hyperf\Signal\SignalHandlerInterface' not found in vendor/hyperf/process/src/Handler/ProcessStopHandler.php on line 17

PHP Fatal error:  Interface 'Symfony\Component\Serializer\SerializerInterface' not found in vendor/hyperf/utils/src/Serializer/Serializer.php on line 46
```

This problem is because in `PHP 7.3`, when obtaining reflection through `sub-process scanning`, if a non-existent `Interface` is implemented in a class, it will cause an `Interface not found` exception to be thrown, while higher versions of `PHP` will not.

The solution is to create the corresponding `Interface` and introduce it normally. The solution to the error mentioned above is to install the corresponding dependent components.

> Of course, it is best to upgrade to version 7.4 or 8.0

```bash
composer require hyperf/signal

composer require symfony/serializer
```

## File System

> Users using File System, please note that those who do not use it can ignore this section.

Starting from version v2.2, Hyperf will simultaneously support the use of `League\Flysystem` component `v1.0` and `v2.0` versions.

If you use `League\Flysystem` version `v2.0`, the corresponding adapter also needs to be adjusted according to the following list. For specific documentation, please refer to the `File System` chapter.

- Alibaba Cloud OSS adapter

```shell
composer require hyperf/flysystem-oss
```

- S3 adapter

```shell
composer require "league/flysystem-aws-s3-v3:^2.0"
composer require hyperf/guzzle
```

- Qiniu adapter

```shell
composer require "overtrue/flysystem-qiniu:^2.0"
```

- Memory adapter

```shell
composer require "league/flysystem-memory:^2.0"
```

- Tencent Cloud COS adapter

> The basic configuration of this adapter is slightly different from the `v2.0` version of the `overtrue/flysystem-cos` component. Please make appropriate modifications according to the latest version configuration.

```shell
composer require "overtrue/flysystem-cos:^4.0"
```
