# Panduan Upgrade 2.0

Versi 2.0 menambahkan banyak fungsi hebat. Jika Anda sudah berinvestasi pada
proyek untuk penggunaan bisnis dan membuat proyek aplikasi versi 1.1 berdasarkan
proyek Skeleton resmi yang disediakan, Anda dapat menyesuaikan proyek Skeleton
Anda sesuai dengan poin-poin berikut: Jika Anda adalah proyek baru, ikuti
dokumentasi untuk membuat proyek baru melalui perintah `composer create-project
hyperf/hyperf-skeleton` untuk menggunakan kode skeleton versi 2.0 yang baru. Jika
saat ini Anda menggunakan versi yang lebih rendah dari 1.1, maka Anda harus
melakukan upgrade ke 1.1 terlebih dahulu dan kemudian upgrade ke 2.0 sesuai
dengan panduan upgrade ini.

## Upgrade Swoole ke 4.5+

Versi 2.0 telah meningkatkan persyaratan versi Swoole minimum dari 4.4+ menjadi
4.5+. Ada beberapa perbedaan dalam detail penggunaan antara kedua versi
tersebut. Hyperf telah menyesuaikan diri dengan perbedaan versi ini pada versi
sebelumnya, sehingga Anda dapat mengabaikannya. Detail perbedaan, meningkatkan
versi Swoole ke 4.5+ terutama untuk mengurangi dampak jangka panjang dari bagasi
historis pada Hyperf. Anda dapat memeriksa versi Swoole di lingkungan saat ini
dengan menjalankan `php --ri swoole`. Anda dapat menyelesaikan upgrade Swoole
sesuai dengan [dokumentasi Swoole](https://wiki.swoole.com/#/environment).

## File Entri Menambahkan Inisialisasi ClassLoader

2.0 telah mengubah logika dasar AOP, sehingga Anda perlu menambahkan satu baris
kode inisialisasi di file entri framework `bin/hyperf.php`. Anda perlu
menambahkan kode `Hyperf\Di\ClassLoader::init();` di baris pertama dari fungsi
anonim entri, seperti berikut:

```php
<?php

ini_set('display_errors', 'on');
ini_set('display_startup_errors', 'on');

error_reporting(E_ALL);
date_default_timezone_set('Asia/Shanghai');

! defined('BASE_PATH') && define('BASE_PATH', dirname(__DIR__, 1));
! defined('SWOOLE_HOOK_FLAGS') && define('SWOOLE_HOOK_FLAGS', SWOOLE_HOOK_ALL);

require BASE_PATH . '/vendor/autoload.php';

// Self-called anonymous function that creates its own scope and keep the global namespace clean.
(function () {
    Hyperf\Di\ClassLoader::init();

    /** @var \Psr\Container\ContainerInterface $container */
    $container = require BASE_PATH . '/config/container.php';

    $application = $container->get(\Hyperf\Contract\ApplicationInterface::class);
    $application->run();
})();

```

Sementara itu, file entri PHPUnit mungkin melakukan hal yang sama, terletak di
`tests/bootstrap.php`, seperti ini:

```php
<?php

declare(strict_types=1);

error_reporting(E_ALL);
date_default_timezone_set('Asia/Shanghai');

! defined('BASE_PATH') && define('BASE_PATH', dirname(__DIR__, 1));
! defined('SWOOLE_HOOK_FLAGS') && define('SWOOLE_HOOK_FLAGS', SWOOLE_HOOK_ALL);

Swoole\Runtime::enableCoroutine(true);

require BASE_PATH . '/vendor/autoload.php';

Hyperf\Di\ClassLoader::init();

$container = require BASE_PATH . '/config/container.php';

$container->get(Hyperf\Contract\ApplicationInterface::class);

```

## Menyesuaikan `composer.json`

Karena penyesuaian logika dasar AOP di versi 2.0, script `init-proxy.sh` telah
dihapus, sehingga Anda perlu menghapus pernyataan pengeksekusian `"init-proxy.sh"`
pada `scripts.post-autoload-dump` dari `composer.json`, dan mengubah perintah di
`post-autoload-dump` menjadi pernyataan `rm -rf runtime/container`.

```json
{
    "scripts": {
        "post-root-package-install": [
            "@php -r \"file_exists('.env') || copy('.env.example', '.env');\""
        ],
        "post-autoload-dump": [
            "rm -rf runtime/container"
        ],
        "analyse": "phpstan analyse --memory-limit 300M -l 0 -c phpstan.neon ./app ./config",
        "cs-fix": "php-cs-fixer fix $1",
        "start": "php ./bin/hyperf.php start",
        "test": "co-phpunit -c phpunit.xml --colors=always"
    }
}

```

## Menyesuaikan Versi Dependensi di composer.json

Karena kita ingin melakukan upgrade ke versi komponen 2.0, dan proyek skeleton
awal bergantung pada komponen versi 1.1.x secara default, kita perlu melakukan
beberapa penyesuaian pada batasan dependensi, dan mengubah semua dependensi
komponen Hyperf awal dari `~1.1.0` menjadi `~2.0.0`, dan Anda juga perlu
mengubah dependensi versi `phpstan/phpstan` menjadi `^0.12`. Setelah perubahan
tersebut, Anda perlu menjalankan `composer update` untuk meng-upgrade dependensi
ke versi 2.0.

## Menyesuaikan Dockerfile

Selama proses pembuatan image Docker, menjalankan perintah `php bin/hyperf.php`
secara aktif akan membantu membuat semua class proxy dan cache pemindaian
annotation yang perlu dibuat sebelumnya, sehingga tidak perlu memindai lagi saat
lingkungan produksi dijalankan, yang dapat sangat mengoptimalkan waktu startup
produksi dan penggunaan memori. Contoh berikut tidak berisi kode Dockerfile yang
tidak diubah.

```dockerfile
ENV TIMEZONE=${timezone:-"Asia/Shanghai"} \
    APP_ENV=prod \
    SCAN_CACHEABLE=(true)

COPY . /opt/www
RUN composer install --no-dev -o && php bin/hyperf.php

EXPOSE 9501

ENTRYPOINT ["php", "/opt/www/bin/hyperf.php", "start"]
```

Bagi pengguna yang tidak melakukan deployment menggunakan `Docker`, perlu dicatat
bahwa sebelum me-restart service, sebaiknya jalankan `php bin/hyperf.php` dan
kemudian restart service untuk mengurangi waktu yang dihabiskan untuk restart.

## Menyesuaikan File Konfigurasi `config/config.php`

Anda perlu menambahkan dua item konfigurasi `app_env` dan `scan_cacheable` ke
dalam konfigurasi `config/config.php`. Contoh kode berikut tidak menyertakan
konten konfigurasi lain yang tidak relevan, seperti yang ditunjukkan di bawah:

```php
<?php

return [
    // Lingkungan produksi menggunakan nilai prod
    'app_env' => env('APP_ENV', 'dev'),
    // Apakah akan memindai cache dengan annotation
    'scan_cacheable' => env('SCAN_CACHEABLE', false),
];
```

Konfigurasi `scan_cacheable` digunakan untuk mengontrol apakah akan menggunakan
annotation untuk memindai cache saat aplikasi dimulai. Ada modifikasi terkait
dalam `Dockerfile` dan `config/config.php` di atas. Ketika nilai konfigurasi ini
adalah `true`, saat proyek dimulai, ia akan menganggap semua class telah dipindai
dan cache serta proxy yang sesuai telah dibuat dengan benar, dan tahap pemindaian
akan dilewati untuk mengoptimalkan waktu startup serta mengurangi overhead memori.

## Mengubah `config/autoload/logger.php`

Karena versi 2.0 telah meningkatkan versi yang bergantung pada Monolog, pada
versi Monolog yang lebih baru, format log default telah berubah. Jika ada
kebutuhan untuk format log tertentu, misalnya kebutuhan untuk terhubung dengan
sistem log sesuai format log, Anda dapat mengubah item konfigurasi `dateFormat`
dari file konfigurasi `config/autoload/logger.php` untuk menjaganya tetap
konsisten dengan versi sebelumnya.

```php
<?php

declare(strict_types=1);

return [
    'default' => [
        'handler' => [
            'class' => Monolog\Handler\StreamHandler::class,
            'constructor' => [
                'stream' => BASE_PATH . '/runtime/logs/hyperf.log',
                'level' => Monolog\Logger::DEBUG,
            ],
        ],
        'formatter' => [
            'class' => Monolog\Formatter\LineFormatter::class,
            'constructor' => [
                'format' => null,
                'dateFormat' => 'Y-m-d H:i:s',
                'allowInlineLineBreaks' => true,
            ],
        ],
        'processors' => [
        ],
    ],
];

```

## Mengubah `config/autoload/exceptions.php`

Versi 2.0 menyesuaikan logika pemrosesan perilaku exception routing HTTP seperti
route tidak ditemukan (404), method request tidak cocok (405), dll., dan diubah
untuk melemparkan class sub-exception dari
`Hyperf\HttpMessage\Exception\HttpException`, kemudian ExceptionHandler digunakan
untuk mengelola exception ini secara seragam dan menangani response yang sesuai,
sehingga pengguna juga dapat memperoleh pengalaman pengembalian response yang
konsisten dengan melemparkan exception yang sesuai. Namun karena ExceptionHandler
adalah mekanisme yang dikelola oleh pengguna, Skeleton default di versi 1.1
mengonfigurasi class `App\Exception\Handler\AppExceptionHandler` untuk menangani
exception tersebut dan mengembalikannya ke klien dengan status code 500. Oleh
karena itu, Anda perlu menggunakan `Hyperf\HttpServer\Exception\Handler\HttpExceptionHandler`
yang disediakan oleh versi 2.0 untuk menangani HttpException di file konfigurasi
`config/autoload/exceptions.php`, dan pastikan posisinya berada di depan
konfigurasi `App\Exception\Handler\AppExceptionHandler`. Contoh konfigurasi
berikut mengabaikan konfigurasi yang tidak relevan, seperti yang ditunjukkan di
bawah:

```php
<?php
return [
    'handler' => [
        'http' => [
            Hyperf\HttpServer\Exception\Handler\HttpExceptionHandler::class,
        ],
    ],
];
```

Setelah Anda menyelesaikan konfigurasi ExceptionHandler, Anda dapat langsung
mengakses route yang tidak ada. Jika status code response adalah `404`, maka dapat
dipahami bahwa modifikasi konfigurasi ini telah selesai.

## Mengubah Client gRPC

Untuk menghindari konflik penamaan dengan bisnis aktual gRPC, versi 2.0 telah
menyesuaikan penamaan fungsi class dasar dari client gRPC.

* `simpleRequest` -> `_simpleRequest`
* `clientStreamRequest` -> `_clientStreamRequest`
* `getGrpcClient` -> `_getGrpcClient`

Selain itu, beberapa method yang tidak seharusnya diekspos telah menjadi private
method. Jika client gRPC Anda melibatkan pemanggilan method di atas, silakan
sesuaikan penamaannya.

## Menghapus Listener Lazy Load DI

Jika proyek Anda menggunakan fungsi lazy loading dari komponen DI, Anda perlu
meregistrasikan listener `Hyperf\Di\Listener\LazyLoaderBootApplicationListener`.
Namun di versi 2.0, listener ini telah dihapus, Anda dapat langsung menggunakan
fungsi ini, jadi jika Anda telah menggunakan fungsi ini sebelumnya, Anda perlu
menghapus registrasi listener tersebut di `config/autoload/listeners.php`;

## Mengikat Hubungan NormalizerInterface

Ketika Anda menggunakan fungsi JSONRPC dan menggunakan library `symfony/serializer`
untuk menyediakan fungsi serialisasi, karena versi 2.0 tidak lagi memetakan
class implementasi dari `Hyperf\Contract\NormalizerInterface` secara otomatis,
Anda perlu menambahkan hubungan pemetaan secara manual, sebagai berikut:

```php
use Hyperf\Utils\Serializer\SerializerFactory;
use Hyperf\Utils\Serializer\Serializer;

return [
    Hyperf\Contract\NormalizerInterface::class => new SerializerFactory(Serializer::class),
];
```

## Menyesuaikan Class Implementasi dari Hyperf\Contract\ProcessInterface

Method `isEnable` pada `Hyperf\Contract\ProcessInterface` menambahkan parameter
`$server`, `isEnable($server): bool`. Semua class implementasi dari ProcessInterface
mengharuskan Anda untuk melakukan beberapa penyesuaian pada method ini.

## Memeriksa File `config/autoload/aspects.php`

Jika Anda telah melakukan beberapa operasi pengurangan (thinning) pada Skeleton
sebelumnya, Anda perlu memeriksa apakah file `config/autoload/aspects.php` ada.
Jika tidak, Anda perlu menambahkan file tersebut dan mengembalikan array kosong.

> Masalah ini akan diperbaiki di 2.0.1, sehingga pemeriksaan ini tidak diperlukan
> lagi di masa mendatang.

## Memeriksa Collector untuk Custom Annotation

Jika Anda menggunakan custom annotation dan menggunakan collector kustom
`Collector` untuk mengumpulkan metadata annotation, Anda perlu mengonfigurasi
`Collector` yang sesuai di `annotations.scan.collectors`. Karena dalam mode
pengembangan, framework akan menentukan apakah file dimodifikasi berdasarkan waktu
modifikasi file, dan kemudian memutuskan apakah akan mengumpulkan kembali metadata
annotation yang sesuai. Jadi, ketika `annotations.scan.collectors` tidak
dikonfigurasi, hal ini akan menyebabkan annotation hanya berfungsi saat `server`
dijalankan pertama kali.

Seperti di layer aplikasi, konfigurasi terletak di file
`config/autoload/annotations.php`, sebagai berikut:

```php
<?php

return [
    'scan' => [
        'collectors' => [
            CustomCollector::class,
        ],
    ],
];
```

Untuk komponen, konfigurasi disediakan oleh ConfigProvider, sebagai berikut:

```php
<?php

return [
    'annotations' => [
        'scan' => [
            'collectors' => [
                CustomCollector::class,
            ],
        ],
    ]
];
```

## Menjalankan Service dan Menguji Interface Akses

Saat menggunakan Swoole 4.5 dan komponen view, jika terjadi masalah interface
404, Anda dapat mencoba menghapus item konfigurasi `static_handler_locations`
pada file `config/autoload/server.php`.

Path di bawah konfigurasi ini akan dianggap sebagai route file statis, sehingga
jika `/` dikonfigurasi, hal itu akan menyebabkan semua interface dianggap sebagai
path file, yang mengakibatkan interface 404.

## Menyelesaikan Upgrade

Pada titik ini, upgrade ke versi 2.0 telah selesai. Namun karena setiap file dasar
Hyperf dapat di-rewrite melalui DI, jika Anda me-rewrite beberapa file internal
framework yang disesuaikan dalam upgrade ini, Anda masih perlu melakukan beberapa
penyesuaian sesuai dengan situasi aktual Anda.

Jika Anda menemui masalah apa pun selama atau setelah upgrade, silakan kunjungi
[Github Issue](https://github.com/hyperf/hyperf/issues) untuk mengirimkan issue
Anda, jelaskan masalah yang Anda temui, dan kami akan mencoba memperbaikinya serta
membantu Anda secepat mungkin.
