# 2.0 Upgrade Guide

Versi 2.0 menambahkan banyak fitur yang kuat. Jika Anda memiliki proyek yang sudah digunakan untuk bisnis dan merupakan proyek versi 1.1 yang dibuat berdasarkan Skeleton resmi, Anda bisa menyesuaikan proyek Skeleton sesuai poin-poin berikut. Jika proyek baru, ikuti dokumentasi dan gunakan `composer create-project hyperf/hyperf-skeleton` untuk membuat proyek baru dengan kode skeleton versi 2.0. Jika versi yang Anda gunakan saat ini lebih rendah dari 1.1, upgrade dulu ke 1.1, lalu ikuti panduan ini untuk upgrade ke 2.0.

## Meningkatkan Swoole ke 4.5+

Versi 2.0 menaikkan persyaratan versi minimum Swoole dari 4.4+ menjadi 4.5+. Ada beberapa perbedaan detail penggunaan antara kedua versi ini. Hyperf telah mengadaptasi perbedaan tersebut di versi sebelumnya, jadi Anda bisa mengabaikannya. Meningkatkan Swoole ke 4.5+ terutama untuk mengurangi dampak jangka panjang dari beban historis pada Hyperf. Cek versi Swoole dengan `php --ri swoole`, dan upgrade sesuai panduan [Dokumentasi Swoole](https://wiki.swoole.com/#/environment).

## Menambahkan Inisialisasi ClassLoader ke Entry File

Versi 2.0 mengubah logika dasar AOP, jadi Anda perlu menambahkan satu baris kode inisialisasi ke file entry framework `bin/hyperf.php`. Tambahkan `Hyperf\Di\ClassLoader::init();` di baris pertama di dalam anonymous function entry, seperti berikut:

```php
<?php

ini_set('display_errors', 'on');
ini_set('display_startup_errors', 'on');

error_reporting(E_ALL);
date_default_timezone_set('Asia/Shanghai');

! defined('BASE_PATH') && define('BASE_PATH', dirname(__DIR__, 1));
! defined('SWOOLE_HOOK_FLAGS') && define('SWOOLE_HOOK_FLAGS', SWOOLE_HOOK_ALL);

require BASE_PATH . '/vendor/autoload.php';

// Anonymous function yang dipanggil sendiri untuk membuat scope sendiri dan menjaga namespace global tetap bersih.
(function () {
    Hyperf\Di\ClassLoader::init();

    /** @var \Psr\Container\ContainerInterface $container */
    $container = require BASE_PATH . '/config/container.php';

    $application = $container->get(\Hyperf\Contract\ApplicationInterface::class);
    $application->run();
})();
```

Pada saat yang sama, file entry PHPUnit mungkin memerlukan penanganan yang sama, dan file tersebut terletak di `tests/bootstrap.php`, seperti yang ditunjukkan di bawah ini:

```php
<?php

declare(strict_types=1);

error_reporting(E_ALL);
date_default_timezone_set('Asia/Shanghai');

! defined('BASE_PATH') && define('BASE_PATH', dirname(__DIR__, 1));
! defined('SWOOLE_HOOK_FLAGS') && define('SWOOLE_HOOK_FLAGS', SWOOLE_HOOK_ALL);

Swoole\Runtime::enableCoroutine(true);

require BASE_PATH . '/vendor/autoload.php';

Hyperf\Di\ClassLoader::init();

$container = require BASE_PATH . '/config/container.php';

$container->get(Hyperf\Contract\ApplicationInterface::class);
```

## Menyesuaikan `composer.json`

Karena penyesuaian logika dasar AOP di versi 2.0, script `init-proxy.sh` telah dihapus. Anda perlu menghapus pernyataan `"init-proxy.sh"` di `scripts.post-autoload-dump` dari `composer.json`, dan mengubah perintah di `post-autoload-dump` menjadi `rm -rf runtime/container`.

```json
{
    "scripts": {
        "post-root-package-install": [
            "@php -r \"file_exists('.env') || copy('.env.example', '.env');\""
        ],
        "post-autoload-dump": [
            "rm -rf runtime/container"
        ],
        "analyse": "phpstan analyse --memory-limit 300M -l 0 -c phpstan.neon ./app ./config",
        "cs-fix": "php-cs-fixer fix $1",
        "start": "php ./bin/hyperf.php start",
        "test": "co-phpunit -c phpunit.xml --colors=always"
    }
}
```

## Menyesuaikan Versi Dependensi di composer.json

Karena Anda ingin meningkatkan ke komponen versi 2.0, dan proyek skeleton asli secara default bergantung pada komponen versi 1.1.x, kita perlu menyesuaikan batasan dependensi, mengubah semua dependensi komponen Hyperf asli dari `~1.1.0` menjadi `~2.0.0`. Pada saat yang sama, Anda juga perlu mengubah dependensi versi `phpstan/phpstan` menjadi `^0.12`. Setelah modifikasi, Anda perlu menjalankan `composer update` untuk meningkatkan dependensi ke versi 2.0.

## Menyesuaikan Dockerfile

Selama proses packaging image Docker, menjalankan `php bin/hyperf.php` akan membantu membuat semua kelas proxy dan cache pemindaian Annotation terlebih dahulu, sehingga tidak perlu pemindaian lagi saat berjalan di lingkungan produksi. Ini bisa mengoptimalkan waktu startup dan penggunaan memori secara signifikan. Contoh berikut tidak menyertakan kode Dockerfile yang tidak dimodifikasi.

```dockerfile
ENV TIMEZONE=${timezone:-"Asia/Shanghai"} \
    APP_ENV=prod \
    SCAN_CACHEABLE=(true)

COPY . /opt/www
RUN composer install --no-dev -o && php bin/hyperf.php

EXPOSE 9501

ENTRYPOINT ["php", "/opt/www/bin/hyperf.php", "start"]
```

Pengguna yang tidak menggunakan `Docker` perlu memperhatikan bahwa sebelum me-restart service, sebaiknya jalankan `php bin/hyperf.php` satu kali terlebih dahulu untuk mengurangi waktu yang dibutuhkan saat restart.

## Menyesuaikan File Konfigurasi config/config.php

Anda perlu menambahkan dua item konfigurasi, `app_env` dan `scan_cacheable`, ke konfigurasi `config/config.php`. Contoh kode di bawah ini tidak menyertakan konten konfigurasi lain yang tidak relevan, seperti yang ditunjukkan di bawah ini:

```php
<?php

return [
    // Lingkungan produksi menggunakan nilai prod
    'app_env' => env('APP_ENV', 'dev'),
    // Apakah akan menggunakan cache pemindaian Annotation
    'scan_cacheable' => env('SCAN_CACHEABLE', false),
];
```

Konfigurasi `scan_cacheable` digunakan untuk mengontrol apakah akan menggunakan Annotation scanning cache saat aplikasi dimulai. Ada modifikasi terkait di `Dockerfile` dan `config/config.php` di atas. Ketika nilai konfigurasi ini `true`, saat proyek dimulai, ia akan mengasumsikan semua kelas telah selesai dipindai dan cache serta proxy yang sesuai telah dihasilkan dengan benar, dan akan melewati tahap pemindaian untuk mengoptimalkan waktu startup dan mengurangi overhead memori.

## Memodifikasi `config/autoload/logger.php`

Karena versi 2.0 meningkatkan versi dependensi Monolog, format log default di versi Monolog yang lebih tinggi telah berubah. Jika ada persyaratan khusus untuk format log, seperti perlu berkomunikasi dengan sistem log, Anda bisa memodifikasi item konfigurasi `dateFormat` di file `config/autoload/logger.php` untuk menjaga konsistensi dengan versi sebelumnya.

```php
<?php

declare(strict_types=1);

return [
    'default' => [
        'handler' => [
            'class' => Monolog\Handler\StreamHandler::class,
            'constructor' => [
                'stream' => BASE_PATH . '/runtime/logs/hyperf.log',
                'level' => Monolog\Logger::DEBUG,
            ],
        ],
        'formatter' => [
            'class' => Monolog\Formatter\LineFormatter::class,
            'constructor' => [
                'format' => null,
                'dateFormat' => 'Y-m-d H:i:s',
                'allowInlineLineBreaks' => true,
            ],
        ],
        'processors' => [
        ],
    ],
];
```

## Memodifikasi `config/autoload/exceptions.php`

Versi 2.0 telah menyesuaikan logika penanganan untuk perilaku abnormal routing HTTP seperti route not found (404) dan request method mismatch (405). Sekarang semua secara seragam melempar subclass exception dari `Hyperf\HttpMessage\Exception\HttpException`, yang kemudian dikelola secara seragam melalui ExceptionHandler. Dengan cara ini, pengguna juga bisa mendapatkan respons yang konsisten dengan melempar exception yang sesuai. Namun, karena ExceptionHandler adalah mekanisme yang dikelola pengguna, dan Skeleton default di versi 1.1 mengonfigurasi `App\Exception\Handler\AppExceptionHandler` untuk menangani exception dan mengembalikan status code 500, Anda perlu mengonfigurasi `Hyperf\HttpServer\Exception\Handler\HttpExceptionHandler` yang disediakan oleh versi 2.0 untuk menangani HttpException di file `config/autoload/exceptions.php`, dan pastikan ia terletak sebelum `App\Exception\Handler\AppExceptionHandler`. Contoh di bawah ini menghilangkan konfigurasi yang tidak relevan:

```php
<?php
return [
    'handler' => [
        'http' => [
            Hyperf\HttpServer\Exception\Handler\HttpExceptionHandler::class,
        ],
    ],
];
```

Setelah menyelesaikan konfigurasi ExceptionHandler, Anda dapat langsung mengakses route yang tidak ada, dan jika status code respons adalah `404`, Anda dapat memahami bahwa modifikasi konfigurasi ini telah selesai.

## Memodifikasi gRPC Client

Untuk menghindari konflik penamaan dengan bisnis gRPC yang sebenarnya, versi 2.0 menyesuaikan penamaan fungsi kelas dasar gRPC client.

* `simpleRequest` -> `_simpleRequest`
* `clientStreamRequest` -> `_clientStreamRequest`
* `getGrpcClient` -> `_getGrpcClient`

Selain itu, beberapa method yang seharusnya tidak diekspos telah menjadi private methods. Jika gRPC client Anda melibatkan pemanggilan method di atas, silakan lakukan penyesuaian penamaan.

## Menghapus DI Lazy Loading Listener

Jika proyek Anda menggunakan fungsi lazy loading dari komponen DI, sebelumnya Anda perlu mendaftarkan listener `Hyperf\Di\Listener\LazyLoaderBootApplicationListener`. Di versi 2.0, listener ini telah dihapus, dan Anda bisa menggunakan fungsi ini secara langsung. Jika sebelumnya Anda menggunakan fungsi ini, hapus pendaftaran listener di `config/autoload/listeners.php`.

## Mengikat Hubungan NormalizerInterface

Ketika Anda menggunakan fungsi JSONRPC dan menggunakan library `symfony/serializer` untuk menyediakan fungsionalitas serialisasi, karena versi 2.0 tidak lagi secara otomatis memetakan kelas implementasi dari `Hyperf\Contract\NormalizerInterface`, Anda perlu menambahkan hubungan pemetaan ini secara manual, sebagai berikut:

```php
use Hyperf\Utils\Serializer\SerializerFactory;
use Hyperf\Utils\Serializer\Serializer;

return [
    Hyperf\Contract\NormalizerInterface::class => new SerializerFactory(Serializer::class),
];
```

## Menyesuaikan Kelas Implementasi dari Hyperf\Contract\ProcessInterface

Method `isEnable` di `Hyperf\Contract\ProcessInterface` menambahkan parameter `$server`, yaitu `isEnable($server): bool`. Semua kelas implementasi dari `ProcessInterface` mengharuskan Anda untuk melakukan beberapa penyesuaian pada method ini.

## Memeriksa File config/autoload/aspects.php

Jika sebelumnya Anda melakukan beberapa operasi penyederhanaan pada Skeleton, Anda perlu memeriksa apakah file `config/autoload/aspects.php` ada. Jika tidak ada, Anda perlu menambahkan file dan mengembalikan array kosong.

> Masalah ini akan diperbaiki di 2.0.1, dan tidak diperlukan pemeriksaan seperti itu selanjutnya.

## Memeriksa Custom Annotation Collector

Jika Anda menggunakan Annotations kustom dan `Collector` kustom untuk mengumpulkan metadata Annotation, Anda perlu mengonfigurasi `Collector` yang sesuai di `annotations.scan.collectors`. Karena dalam mode pengembangan, ini akan menentukan apakah file telah dimodifikasi berdasarkan waktu modifikasi, lalu memutuskan apakah akan mengumpulkan ulang metadata Annotation. Oleh karena itu, ketika `annotations.scan.collectors` tidak dikonfigurasi, Annotation hanya akan berlaku saat `server` pertama kali dijalankan.

Misalnya, di lapisan aplikasi, konfigurasi ini terletak di file `config/autoload/annotations.php`, sebagai berikut:

```php
<?php

return [
    'scan' => [
        'collectors' => [
            CustomCollector::class,
        ],
    ],
];
```

Misalnya, di dalam komponen, konfigurasi ini disediakan oleh `ConfigProvider`, sebagai berikut:

```php
<?php

return [
    'annotations' => [
        'scan' => [
            'collectors' => [
                CustomCollector::class,
            ],
        ],
    ]
];
```

## Memulai Service dan Menguji Akses Interface

Jika terjadi masalah interface 404 saat menggunakan Swoole versi 4.5 dan komponen view, Anda dapat mencoba menghapus item konfigurasi `static_handler_locations` di file `config/autoload/server.php`.

Path di bawah konfigurasi ini semuanya akan dianggap sebagai static file routes. Oleh karena itu, jika `/` dikonfigurasi, semua interface akan dianggap sebagai file path, yang mengakibatkan interface 404.

## Menyelesaikan Upgrade

Sejauh ini, upgrade ke versi 2.0 telah selesai. Namun, karena setiap file dasar Hyperf bisa ditulis ulang melalui DI, jika Anda telah menulis ulang beberapa file framework internal yang disesuaikan dalam upgrade ini, Anda masih perlu melakukan penyesuaian sesuai situasi aktual.

Jika Anda mengalami masalah selama atau setelah upgrade, silakan pergi ke [Github Issue](https://github.com/hyperf/hyperf/issues) untuk mengirimkan issue Anda, menjelaskan masalah yang Anda temui, dan kami akan membantu Anda menyelesaikannya sesegera mungkin.
