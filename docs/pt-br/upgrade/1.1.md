# Guia de atualização 1.1

A versão 1.1 adiciona muitas funcionalidades, mas algumas mudanças também envolvem ajustes no Skeleton e alterações estruturais em itens de configuração. Se você já investiu em um projeto para uso em produção e criou um projeto de aplicação 1.0 com base no Skeleton oficial, então você pode ajustar seu projeto skeleton de acordo com os pontos a seguir. Se você for um novo projeto, siga a documentação para criar um novo projeto com o comando `composer create-project hyperf/hyperf-skeleton` para usar a nova estrutura do skeleton.

## Atualizar Swoole para 4.4+

A versão 1.1 elevou a versão mínima exigida do Swoole de 4.3+ para 4.4+. Há alguns detalhes de uso entre essas duas versões. O Hyperf já se adaptou em versões anteriores, então não há necessidade de os usuários do Hyperf ignorarem as diferenças. Aumentamos a versão mínima do Swoole principalmente para reduzir nosso fardo histórico, e como o Swoole 4.4 é o LTS (Long Term Support Version), isso também significa ser mais estável e confiável.

O Hyperf detecta a versão do Swoole na inicialização, mas para unificar melhor as dependências em versões do Swoole em todos os lugares, recomendamos que você altere a dependência do Swoole no `composer.json` para `"ext-swoole": " >=4.4"`.

## Adicionar a constante SWOOLE_HOOK_FLAGS

Adicione uma linha de definição de constante ao arquivo de entrada `bin/hyperf.php` da aplicação e ao arquivo de entrada `test/bootstrap.php` do teste unitário, como a seguir:

```php
! defined('SWOOLE_HOOK_FLAGS') && define('SWOOLE_HOOK_FLAGS', SWOOLE_HOOK_ALL);
```

Referência: [Entry file reference](https://github.com/hyperf/hyperf-skeleton/blob/70062b7bbf29e23cda2f30680e02aa3b26ebd6f7/bin/hyperf.php#L11) [Single test entry file reference](https://github.com/ hyperf/hyperf-skeleton/blob/70062b7bbf29e23cda2f30680e02aa3b26ebd6f7/test/bootstrap.php#L20)

## Mover o arquivo config/dependencies.php e ajustar a estrutura

Mova `config/dependencies.php` → `config/autoload/dependencies.php` e remova o primeiro nível `dependencies` do arquivo de configuração, como a seguir:

Estrutura de arquivo do 1.0:

```php
<?php
// documento config/dependencies.php

return [
    'dependencies' => [
        FooInterface::class => Foo::class
    ],
];
```

Estrutura de arquivo do 1.1:

```php
<?php
// documento config/autoload/dependencies.php

return [
    FooInterface::class => Foo::class
];
```

## Ajustar o conteúdo do arquivo config/container.php

Como a versão 1.1 ajustou a localização e a estrutura do arquivo `dependencies.php`, também precisamos ajustar o arquivo `config/container.php` para que o container de injeção de dependência rode corretamente. O `config/container.php` oferece uma forma mais simples de escrita; o `DefinitionSourceFactory` agrega muitos comportamentos padrão. Você só precisa substituir o conteúdo do arquivo `config/container.php` pelo seguinte conteúdo:

> A função de cache de scanning de anotações é habilitada por padrão. Você pode modificar o primeiro parâmetro do `DefinitionSourceFactory` para desabilitar esta função.

```php
<?php
/**
 * Inicializa um container de injeção de dependência que implementa PSR-11 e retorna o container.
 */
declare(strict_types=1);

use Hyperf\Di\Container;
use Hyperf\Di\Definition\DefinitionSourceFactory;
use Hyperf\Utils\ApplicationContext;
use Psr\Container\ContainerInterface;

$container = new Container((new DefinitionSourceFactory(true))());
if (! $container instanceof ContainerInterface) {
    throw new RuntimeException('The dependency injection container is invalid.');
}
return ApplicationContext::setContainer($container);
```

## Ajustar o WebSocket controller

Como a versão 1.1 ajustou as restrições de entrada de `onMessage` e `onOpen`, você precisa modificá-las manualmente para `Swoole\WebSocket\Server`. O código é o seguinte:

```php
<?php
declare(strict_types=1);

namespace App\Controller;

use Hyperf\Contract\OnMessageInterface;
use Hyperf\Contract\OnOpenInterface;
use Swoole\Http\Request;
use Swoole\Websocket\Frame;
use Swoole\WebSocket\Server as WebSocketServer;

class WebSocketController implements OnMessageInterface, OnOpenInterface
{
    public function onMessage(WebSocketServer $server, Frame $frame): void
    {
    }

    public function onOpen(WebSocketServer $server, Request $request): void
    {
    }
}
```

## Ajustar o ConfigProvider do componente customizado

No 1.0, `scan.path` foi ajustado para `annotations.scan.path` no 1.1. Você precisa modificar a classe ConfigProvider de todos os componentes customizados para se adaptar a essa mudança. Por exemplo: se seus componentes customizados não envolvem scanning de anotações, este ajuste pode ser ignorado. Exemplos:

Estrutura do ConfigProvider no 1.0:

```php
class ConfigProvider
{

    public function __invoke(): array
    {
        return [
            'scan' => [
                'paths' => [
                    __DIR__,
                ],
            ],
        ];
    }
}
```

Estrutura do ConfigProvider no 1.1:

```php
class ConfigProvider
{

    public function __invoke(): array
    {
        return [
            'annotations' => [
                'scan' => [
                    'paths' => [
                        __DIR__,
                    ],
                ],
            ],
        ];
    }
}
```

## Ajustar o idioma padrão de localization

Se você já usou o componente [hyperf/translation](https://github.com/hyperf/translation) antes, então você precisa verificar o item de configuração `locale` no arquivo `config/autoload/translation.php`. Por exemplo, se for `zh-CN`, ele precisa ser alterado para `zh_CN`. Na versão 1.1, padronizamos o valor desta configuração.

## Ajustar dependências do composer.json

Como queremos atualizar para a versão 1.1 dos componentes e o skeleton original depende por padrão da versão 1.0.x dos componentes, precisamos ajustar os constraints das dependências: mudar as dependências de todos os componentes Hyperf originais de `~1.0.0 ` para `~1.1.0`. Após a modificação, você precisa executar `composer update` para atualizar as dependências para a versão 1.1.

Todas as dependências do Hyperf precisam ser atualizadas para 1.1 para estarem disponíveis, pois o 1.1 ajusta o mecanismo ConfigProvider para adaptação dos componentes.

## Concluir a atualização

Neste ponto, a atualização para 1.1 foi concluída. Porém, como cada arquivo interno do Hyperf pode ser reescrito via DI, se você reescreveu alguns arquivos internos do framework ajustados nesta atualização, você ainda precisará ajustar essas reescritas de acordo com a sua situação real.

Se você encontrar algum problema durante ou depois da atualização, vá em [Github Issue](https://github.com/hyperf/hyperf/issues) para enviar seu issue, explique o problema encontrado e tentaremos corrigir o mais rápido possível para ajudar você.
