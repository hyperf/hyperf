# Guia de atualização 3.1

- A versão 3.1 altera principalmente a versão mínima do `PHP` para `8.1` e a versão mínima do `Swoole` para `5.0`.
- Introduz o framework de testes `Pest`
- Adiciona `hyperf/helper` e migra as `None namespace helper functions` de `Hyperf\Utils` para `hyperf/helper`.
- Altera a forma como `hyperf/config` carrega arquivos de configuração em múltiplos níveis para suportar a sintaxe `. `; por exemplo: `config('a.c')`

## Mudanças no pacote Utils

As helper functions no pacote utils original não tinham namespaces e poderiam entrar em conflito com funções em outros pacotes do composer; então elas foram removidas e substituídas no 3.1.

Há duas formas de lidar com isso:

- Se nenhum outro pacote for introduzido na aplicação, isso causará conflitos de funções; então você pode instalar `hyperf/helper` diretamente, e o pacote `helper` fornece helper functions sem namespaces, o mesmo do pacote `utils` original.
- Introduza namespaces onde as helper functions são chamadas e refatore usando `rector` [refactoring docs here 🔎](https://github.com/orgs/hyperf/discussions/5635)

## Mudanças no carregamento de arquivos de configuração

No 3.0, as configurações existiam com `config file name` como chave; já no 3.1, as configs multi-nível do `hyperf/config` são carregadas com `relative directory.Config file name` como chave.

Suporte à sintaxe `. `, como `config('a.c')`.

Sobre isso, é importante observar que os dois casos abaixo devem ser tratados no projeto original:

- Em um projeto 3.0 foi criado um subdiretório em `config/autoload`, por exemplo `config/autoload/sub/a.php`; então o `config('a')` original precisa ser alterado para `config('sub.a')`.
- Nomes de arquivos de configuração em projetos 3.0 contêm `. `, como `config/autoload/a.b.php`, e ao usar `config('a')` isso retornará a estrutura a seguir; e se você também tiver um arquivo `config/autoload/a.php`, você também obterá o resultado após mesclar os itens de configuração.

```php
return [
    'a.php config key' => 'a.php config value',
    'b' => [
        'a.b.php config key' => 'a.b.php config value',
    ]
];
```

## Outras mudanças

1. o componente `hyperf/metirc` não instala mais a dependência `prometheus` por padrão, então desenvolvedores que precisem usar o engine `prometheus` precisam executar o comando a seguir para carregar as dependências:

```bash
composer require promphp/prometheus_client_php
```

2. Mudanças em `withHeaders` para `Request` e `Response`

Antes do v3.1, o comportamento mantinha os dados originais, semelhante ao `array_merge`. Após a versão v3.1, todos os `headers` serão substituídos diretamente.

```php
<?php
// antes do v3.1
$request->withHeader('foo', 1)->withHeader('bar', 2); // ['foo' => [1], 'bar' => [2]]
$request->withHeader('foo', 1)->withHeader('foo', 2); // ['foo' => [2]]
$request->withAddedHeader('foo', 1)->withAddedHeader('foo', 2); // ['foo' => [1, 2]]

$request->withHeaders(['foo' => 1])->withHeaders(['bar' => 2]); // ['foo' => [1], 'bar' => [2]]

// depois do v3.1
$request->withHeader('foo', 1)->withHeader('bar', 2); // ['foo' => [1], 'bar' => [2]]
$request->withHeader('foo', 1)->withHeader('foo', 2); // ['foo' => [2]]
$request->withAddedHeader('foo', 1)->withAddedHeader('foo', 2); // ['foo' => [1, 2]]

$request->withHeaders(['foo' => 1])->withHeaders(['bar' => 2]); // ['bar' => [2]]

$request->setHeader('foo', 1)->setHeader('bar', 2); // ['foo' => [1], 'bar' => [2]]
$request->setHeader('foo', 1)->setHeader('foo', 2); // ['foo' => [2]]
$request->addHeader('foo', 1)->addHeader('foo', 2); // ['foo' => [1, 2]]

$request->setHeaders(['foo' => 1])->setHeaders(['bar' => 2]); // ['bar' => [2]]
```

3. O valor de retorno do método `consumeMessage` do consumidor `AMQP` foi ajustado para o enum Result

```php
<?php
// antes do v3.1
public function consumeMessage($data, AMQPMessage $message): string
{
    return Result::ACK;
}
// depois do v3.1
public function consumeMessage($data, AMQPMessage $message): Result
{
    return Result::ACK;
}
```
