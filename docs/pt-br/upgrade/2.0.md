# Guia de atualização 2.0

A versão 2.0 adicionou muitas funcionalidades poderosas. Se você já investiu em um projeto para uso em produção e criou um projeto de aplicação da versão 1.1 com base no Skeleton oficial, você pode ajustar seu projeto Skeleton de acordo com os pontos a seguir. Se você for um projeto novo, siga a documentação para criar um novo projeto via o comando `composer create-project hyperf/hyperf-skeleton` para usar a nova versão 2.0 do código do skeleton. Se você está usando atualmente uma versão menor que 1.1, então precisa primeiro atualizar para 1.1 e depois atualizar para 2.0 seguindo este guia.

## Atualizar Swoole para 4.5+

A versão 2.0 elevou o requisito mínimo do Swoole de 4.4+ para 4.5+. Existem algumas diferenças de detalhes de uso entre as duas versões. O Hyperf já se adaptou a essa diferença em versões anteriores, então você pode ignorar. No entanto, atualizar o Swoole para 4.5+ é principalmente para reduzir o impacto de longo prazo do legado histórico no Hyperf. Você pode verificar a versão do Swoole no ambiente atual executando `php --ri swoole`. Você pode concluir a atualização do Swoole de acordo com a [documentação do Swoole](https://wiki.swoole.com/#/environment).

## Adicionar inicialização do ClassLoader no arquivo de entrada

A versão 2.0 alterou a lógica subjacente do AOP, então você precisa adicionar uma linha de código de inicialização no arquivo de entrada do framework `bin/hyperf.php`. Você precisa adicionar o código `Hyperf\Di\ClassLoader::init();` como a primeira linha da função anônima de entrada, como a seguir:

```php
<?php

ini_set('display_errors', 'on');
ini_set('display_startup_errors', 'on');

error_reporting(E_ALL);
date_default_timezone_set('Asia/Shanghai');

! defined('BASE_PATH') && define('BASE_PATH', dirname(__DIR__, 1));
! defined('SWOOLE_HOOK_FLAGS') && define('SWOOLE_HOOK_FLAGS', SWOOLE_HOOK_ALL);

require BASE_PATH . '/vendor/autoload.php';

// Função anônima autoexecutável que cria seu próprio escopo e mantém o namespace global limpo.
(function () {
    Hyperf\Di\ClassLoader::init();

    /** @var \Psr\Container\ContainerInterface $container */
    $container = require BASE_PATH . '/config/container.php';

    $application = $container->get(\Hyperf\Contract\ApplicationInterface::class);
    $application->run();
})();

```

Ao mesmo tempo, o arquivo de entrada do PHPUnit pode fazer o mesmo, localizado em `tests/bootstrap.php`, assim:

```php
<?php

declare(strict_types=1);

error_reporting(E_ALL);
date_default_timezone_set('Asia/Shanghai');

! defined('BASE_PATH') && define('BASE_PATH', dirname(__DIR__, 1));
! defined('SWOOLE_HOOK_FLAGS') && define('SWOOLE_HOOK_FLAGS', SWOOLE_HOOK_ALL);

Swoole\Runtime::enableCoroutine(true);

require BASE_PATH . '/vendor/autoload.php';

Hyperf\Di\ClassLoader::init();

$container = require BASE_PATH . '/config/container.php';

$container->get(Hyperf\Contract\ApplicationInterface::class);

```

## Ajustar `composer.json`

Devido ao ajuste da lógica subjacente do AOP na versão 2.0, o script `init-proxy.sh` foi removido. Então você precisa remover a instrução `"init-proxy.sh"` de `scripts.post-autoload-dump` no `composer.json` e modificar o comando em `post-autoload-dump` para a instrução `rm -rf runtime/container`.

```json
{
    "scripts": {
        "post-root-package-install": [
            "@php -r \"file_exists('.env') || copy('.env.example', '.env');\""
        ],
        "post-autoload-dump": [
            "rm -rf runtime/container"
        ],
        "analyse": "phpstan analyse --memory-limit 300M -l 0 -c phpstan.neon ./app ./config",
        "cs-fix": "php-cs-fixer fix $1",
        "start": "php ./bin/hyperf.php start",
        "test": "co-phpunit -c phpunit.xml --colors=always"
    }
}

```

## Ajustar a versão das dependências no composer.json

Como queremos atualizar para a versão 2.0 dos componentes, e o skeleton original depende por padrão da versão 1.1.x dos componentes, precisamos ajustar os constraints de dependências: alterar as dependências de todos os componentes Hyperf originais de `~1.1.0` para `~2.0.0`. Você também precisa modificar a dependência de versão de `phpstan/phpstan` para `^0.12`. Após a modificação, execute `composer update` para atualizar as dependências para a versão 2.0.

## Ajustar Dockerfile

Durante o processo de build da imagem Docker, executar ativamente o comando `php bin/hyperf.php` ajuda a criar antecipadamente todas as classes proxy e os caches de scanning de anotações que precisam ser gerados, de modo que não seja necessário escanear novamente quando o ambiente de produção iniciar. Isso pode otimizar bastante o tempo de inicialização em produção e o uso de memória. Os exemplos abaixo não contêm código do Dockerfile que não foi modificado.

```dockerfile
ENV TIMEZONE=${timezone:-"Asia/Shanghai"} \
    APP_ENV=prod \
    SCAN_CACHEABLE=(true)

COPY . /opt/www
RUN composer install --no-dev -o && php bin/hyperf.php

EXPOSE 9501

ENTRYPOINT ["php", "/opt/www/bin/hyperf.php", "start"]
```

Para usuários que não fazem deploy via `Docker`, vale notar que, antes de reiniciar o serviço, é melhor executar `php bin/hyperf.php` e então reiniciar o serviço, para reduzir o tempo gasto no reinício.

## Ajustar o arquivo de configuração config/config.php

Você precisa adicionar dois itens de configuração, `app_env` e `scan_cacheable`, na configuração `config/config.php`. O exemplo abaixo omite conteúdos irrelevantes, conforme mostrado:

```php
<?php

return [
    // O ambiente de produção usa o valor prod
    'app_env' => env('APP_ENV', 'dev'),
    // Se deve escanear o cache com anotações
    'scan_cacheable' => env('SCAN_CACHEABLE', false),
];
```

A configuração `scan_cacheable` é usada para controlar se o cache de scanning de anotações será usado quando a aplicação iniciar. Há modificações relacionadas no `Dockerfile` e no `config/config.php` acima. Quando o valor dessa configuração é `true`, ao iniciar o projeto ele entenderá que todas as classes já foram escaneadas e que os caches e proxies correspondentes foram gerados corretamente; então a fase de scanning será ignorada para otimizar o tempo de inicialização e reduzir overhead de memória.

## Modificar `config/autoload/logger.php`

Como a versão 2.0 atualizou a versão do Monolog usada, em versões mais novas do Monolog o formato padrão do log mudou. Se houver requisitos de formato de logs — como a necessidade de integração com um sistema de logs de acordo com o formato — você pode modificar o item de configuração `dateFormat` do arquivo `config/autoload/logger.php` para mantê-lo consistente com a versão anterior.

```php
<?php

declare(strict_types=1);

return [
    'default' => [
        'handler' => [
            'class' => Monolog\Handler\StreamHandler::class,
            'constructor' => [
                'stream' => BASE_PATH . '/runtime/logs/hyperf.log',
                'level' => Monolog\Logger::DEBUG,
            ],
        ],
        'formatter' => [
            'class' => Monolog\Formatter\LineFormatter::class,
            'constructor' => [
                'format' => null,
                'dateFormat' => 'Y-m-d H:i:s',
                'allowInlineLineBreaks' => true,
            ],
        ],
        'processors' => [
        ],
    ],
];

```

## Modificar `config/autoload/exceptions.php`

A versão 2.0 ajustou a lógica de tratamento de exceções de roteamento HTTP, como route not found (404), request method mismatch (405), etc., e mudou para lançar uma subclasse de exceção de `Hyperf\HttpMessage\Exception\HttpException`. Em seguida, o ExceptionHandler é usado para gerenciar essas exceções de forma unificada e tratar as respostas correspondentes, de modo que usuários também possam obter uma experiência consistente de retorno ao lançar as exceções apropriadas. Porém, como ExceptionHandler é um mecanismo gerenciado pelo usuário, o Skeleton 1.1 configura por padrão uma classe `App\Exception\Handler\AppExceptionHandler` para tratar a exceção e retornar ao cliente com status 500. Então, você precisa configurar o `Hyperf\HttpServer\Exception\Handler\HttpExceptionHandler` fornecido na versão 2.0 no arquivo `config/autoload/exceptions.php`, e garantir que ele esteja antes da configuração de `App\Exception\Handler\AppExceptionHandler`. O exemplo abaixo omite configurações extras:

```php
<?php
return [
    'handler' => [
        'http' => [
            Hyperf\HttpServer\Exception\Handler\HttpExceptionHandler::class,
        ],
    ],
];
```

Após concluir a configuração do ExceptionHandler, você pode acessar diretamente uma rota que não existe. Se o status code da resposta for `404`, pode-se entender que a modificação desta configuração foi concluída.

## Modificar o client gRPC

Para evitar conflitos de nome com a lógica de negócio real do gRPC, a versão 2.0 ajustou o nome das funções da classe base do client gRPC:

* `simpleRequest` -> `_simpleRequest`
* `clientStreamRequest` -> `_clientStreamRequest`
* `getGrpcClient` -> `_getGrpcClient`

Além disso, alguns métodos que não deveriam ser expostos se tornaram métodos private. Se o seu client gRPC envolve a invocação dos métodos acima, ajuste os nomes.

## Remover o listener de lazy load do DI

Se o seu projeto usa a funcionalidade de lazy loading dos componentes de DI, você precisa registrar o listener `Hyperf\Di\Listener\LazyLoaderBootApplicationListener`, mas na versão 2.0 este listener foi removido. Você pode usar esta funcionalidade diretamente; então, se você usou isso antes, precisa remover o registro do listener em `config/autoload/listeners.php`.

## Vincular o relacionamento NormalizerInterface

Ao usar JSONRPC e a biblioteca `symfony/serializer` para fornecer funções de serialização, como a versão 2.0 não mapeia mais automaticamente a classe de implementação de `Hyperf\Contract\NormalizerInterface`, você precisa adicionar manualmente o relacionamento de mapeamento, como a seguir:

```php
use Hyperf\Utils\Serializer\SerializerFactory;
use Hyperf\Utils\Serializer\Serializer;

return [
    Hyperf\Contract\NormalizerInterface::class => new SerializerFactory(Serializer::class),
];
```

## Ajustar a classe de implementação de Hyperf\Contract\ProcessInterface

O método `isEnable` em `Hyperf\Contract\ProcessInterface` adicionou um parâmetro `$server`, `isEnable($server): bool`. Todas as classes de implementação de ProcessInterface precisam fazer ajustes neste método.

## Verificar o arquivo config/autoload/aspects.php

Se você fez alguma operação de "thinning" no Skeleton antes, precisa verificar se o arquivo `config/autoload/aspects.php` existe. Se não existir, você precisa adicionar o arquivo e retornar um array vazio.

> Este problema será corrigido no 2.0.1, então essa verificação não será necessária no futuro.

## Verificar o collector para anotações customizadas

Se você usa anotações customizadas e um collector customizado `Collector` para coletar metadados de anotações, você precisa configurar o `Collector` correspondente em `annotations.scan.collectors`. Isso porque, no modo de desenvolvimento, será usado o tempo de modificação do arquivo para determinar se o arquivo foi modificado e, então, decidir se deve coletar novamente os metadados de anotações correspondentes. Portanto, quando `annotations.scan.collectors` não está configurado, isso fará com que as anotações só tenham efeito quando o `server` for iniciado pela primeira vez.

Na camada de aplicação, a configuração fica no arquivo `config/autoload/annotations.php`, como a seguir:

```php
<?php

return [
    'scan' => [
        'collectors' => [
            CustomCollector::class,
        ],
    ],
];
```

Para componentes, a configuração é fornecida por ConfigProvider, assim:

```php
<?php

return [
    'annotations' => [
        'scan' => [
            'collectors' => [
                CustomCollector::class,
            ],
        ],
    ]
];
```

## Iniciar o serviço e testar a interface de acesso

Ao usar Swoole 4.5 e o componente de view, se houver um problema de 404 nas interfaces, você pode tentar apagar o item de configuração `static_handler_locations` no arquivo `config/autoload/server.php`.

Paths sob essa configuração serão considerados rotas de arquivos estáticos; então, se `/` estiver configurado, isso fará com que todas as interfaces sejam consideradas paths de arquivos, resultando em 404.

## Concluir a atualização

Neste ponto, a atualização para a versão 2.0 foi concluída. Porém, como cada arquivo subjacente do Hyperf pode ser reescrito via DI, se você reescrever alguns arquivos internos do framework ajustados nesta atualização, você ainda precisará ajustar essas reescritas de acordo com a sua situação real.

Se você encontrar algum problema durante ou depois da atualização, vá em [Github Issue](https://github.com/hyperf/hyperf/issues) para enviar seu issue, explique o problema encontrado e tentaremos corrigir o mais rápido possível para ajudar você.
