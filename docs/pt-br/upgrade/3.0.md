# Guia de atualização 3.0

- A versão 3.0 modifica principalmente o `PHP`; a versão mínima é `8.0`
- O framework remove `Doctrine Annotations` e usa `PHP8 Attributes` em seu lugar
- O framework adiciona um grande número de restrições de tipos em variáveis membro

## Converter todas as anotações

**Note: esta etapa só pode ser executada na versão 2.2**

O script a seguir converte todas as `Doctrine Annotations` para `PHP8 Attributes`.

```shell
composer require hyperf/code-generator
php bin/hyperf.php code:generate -D app
```

## Modificar a versão de todos os componentes Hyperf

Basta alterar `hyperf/*` no `composer.json` para `3.0.*`.

> hyperf/engine não segue o número de versão do framework, então não é necessário modificá-lo; apenas garanta que ele esteja na versão ^2.1.0

Depois disso, basta executar `composer update -o`, e a atualização pode ser concluída normalmente.

## Atualizar o model do banco de dados

Como a classe base do model adiciona suporte a tipos em variáveis membro, você precisa usar o script a seguir para atualizá-lo para a nova versão.

```shell
composer require hyperf/code-generator
php vendor/bin/regenerate-models.php $PWD/app/Model
```

## Logger

A versão 3.x de `monolog/monolog` usa novos recursos do PHP8.1, então algumas classes precisam ser modificadas de forma específica.

Modifique `array $record` para `array|LogRecord $record` para ser compatível com a versão 3.x. O exemplo é o seguinte:

```php
<?php

declare(strict_types=1);

namespace App\Kernel\Log;

use Hyperf\Context\Context;
use Hyperf\Coroutine\Coroutine;
use Monolog\LogRecord;
use Monolog\Processor\ProcessorInterface;

class AppendRequestIdProcessor implements ProcessorInterface
{
    public const REQUEST_ID = 'log.request.id';

    public function __invoke(array|LogRecord $record)
    {
        $record['extra']['request_id'] = Context::getOrSet(self::REQUEST_ID, uniqid());
        $record['extra']['coroutine_id'] = Coroutine::id();
        return $record;
    }
}

```

##Command

Após a versão `3.0`, a linha de comando tem um event listener habilitado por padrão, então quando um listener escuta o evento `Command` e executa `AMQP` ou outra lógica de multiplexação, o processo não consegue sair.

Solução:

- método um:

Ao executar o comando, adicione a opção `--disable-event-dispatcher`.

- Método Dois:

Adicionar listener

```php
<?php

declare(strict_types=1);

namespace App\Listener;

use Hyperf\Command\Event\AfterExecute;
use Hyperf\Coordinator\Constants;
use Hyperf\Coordinator\CoordinatorManager;
use Hyperf\Event\Annotation\Listener;
use Hyperf\Event\Contract\ListenerInterface;

#[Listener]
class ResumeExitCoordinatorListener implements ListenerInterface
{
    public function listen(): array
    {
        return [
            AfterExecute::class,
        ];
    }

    public function process(object $event): void
    {
        CoordinatorManager::until(Constants::WORKER_EXIT)->resume();
    }
}
```

## Iniciar o servidor

Em seguida, basta iniciar o servidor e você poderá ver o código incompatível e modificá-lo um a um.

- Tipos foram adicionados em variáveis membro de AMQP Consumer e Producer
- O método `process` do Listener adiciona o tipo de retorno `void`
- A anotação `#[CircuitBreaker]` mudou o parâmetro `$timeout` para `$options.timeout`

## GRPC

O framework modifica o `Http status` retornado pelo `GRPC Server` para 200 de acordo com a especificação `GRPC`, e o `GRPC Server` retorna o `status code` correspondente. Faça upgrade para a versão `3.x`; caso contrário, a outra ponta não conseguirá parsear o erro normalmente quando a requisição estiver anormal.
